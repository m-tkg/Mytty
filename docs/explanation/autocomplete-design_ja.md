# オートコンプリートの設計

English version is [autocomplete-design.md](autocomplete-design.md).

Mytty のインライン オートコンプリートは実装済みの機能としてすでに出荷されている
(`README.md` の「Local autocomplete」の記述と、実装本体である
`Sources/MyTTYApp/TerminalAutocomplete.swift` /
`TerminalAutocompleteCoordinator.swift` を参照)。このページはその設計判断を
説明する。出荷された機能が、当初の実現可能性調査(feasibility study)が
描いていた計画よりも意図的に小さくなっている点も含めて振り返る。

## 実際に出荷されたもの

Mytty のオートコンプリートは、一度に一つだけのインライン候補をカーソン直後に
薄いゴーストテキストとして表示し、Tab で確定する。候補の出所は 2 つだけに
絞られている。ペインが「成功したコマンド」を学習していく中で、直近 200 件の
うちのどれかの先頭部分と一致する入力があれば、続きを提案する。もう一つは
単一のヒューリスティックで、`mkdir <path>` が成功したあとに `cd <path>` を
次のコマンドとして提案する。

候補パネルは存在せず、コマンド・フラグ・引数の静的データベースも持たない。
アプリを再起動すればそれまでに学習したコマンド履歴も消える。ペインが生きている
間だけメモリ上に保持される仕組みだ。これは、以前の実現可能性調査
(この文書を書いている時点で、そこにあった「保留中」という記載自体が
すでに古くなっていた)が描いていた、Fig 互換の多段階システムに比べる
とかなり小さい。その計画と実際に出荷されたものとのギャップこそが、ここで
説明する価値のある設計の物語であり、単なる未達として詫びるべきものではない。

## シェルブリッジなしで入力バッファを追跡する

実現可能性調査が出した中心的な結論は、libghostty のキーイベントと画面上の
テキストだけからシェルの入力バッファを再構成するのは一般には信頼できない、
というものだった。シェルが履歴、補完プラグイン、複数行編集、ペースト、
vi モード、カーソル移動を自前で処理し始めた瞬間に、その再構成は正しさの
根拠を失う。そこで提案されていたのは zsh の ZLE(Zsh Line Editor)と連携し、
シェル本体が持つ実際の `BUFFER` と `CURSOR` を、サーフェスごとに常設した
IPC チャネル経由で報告させるという方式だった。これなら Mytty 側は推測ではなく
権威あるデータを直接受け取れる。

そのブリッジは結局作られなかった。代わりに出荷されたのは、実現可能性調査が
「却下すべき方式」として挙げていたものに近い。ただし、バッファの内容が
わからなくなった瞬間にそれを正直に認めて動作を止めることで安全に倒して
ある。`TerminalAutocomplete.swift` の `TerminalAutocompleteSession` は、
`TerminalAutocompleteEventMapper` を通してキーイベントを監視し、現在の行の
内容をそれなりに再現しようとする。印字可能な文字はローカルの `currentInput`
文字列に追記され、Delete は末尾を削り、Return/Enter でコマンドとして確定
される。矢印キー、Control/Option 修飾を伴うナビゲーション、IME 変換中の
マークテキストなど、正しく追跡できないものを検知すると、`inputIsReliable`
フラグを `false` にして、行がリセットされる(Return、または Control-U の
ような明示的なリセット)まで候補提案自体を止める。誤った推測が画面に出る
ことはなく、セッションはただ黙るだけだ。

これが成立するのは、この機能の被害範囲が設計上そもそも小さいからだ。候補は
カーソル位置に表示される 1 行のゴーストテキストにすぎず、Tab 以外のどんな
キー入力も候補を消すか更新し、ユーザーが Tab を押さない限り端末には何も
注入されない。ときどき間違うシャドウバッファでも、間違えたときの結果が
「候補が消えるだけ」であれば許容できる。もしそれが端末を勝手に編集する
機能だったら、この程度の不確実性は許容できなかったはずだ。この許容度の
高さは、シェル固有のブリッジが結局不要だったことの理由でもある。出荷された
仕組みはペインが zsh を動かしているか bash を動かしているか fish を動かして
いるかを一切気にしない。そもそもシェルと通信していないからだ。AppKit 層で
NSEvent を読んでいるだけであり、当初の段階計画(まず zsh、その後 bash と
fish を後の段階で追加)は今実際に存在するものには当てはまらない。拡張すべき
シェル固有の統合そのものが最初から存在しないからだ。

## コマンド完了の検知

コマンドがいつ終わったか、終了コードは何かを知ることは、それを履歴として
学習するかどうか、`mkdir` → `cd` の追加提案を出すかどうかを判断するために
必要になる。Mytty 自身は OSC 133 のシェル統合マークを直接パースしていない。
libghostty がすでに内部でそれを追跡していて、終了コードと実行時間を持つ
構造化された `GHOSTTY_ACTION_COMMAND_FINISHED` アクションとして公開して
くれる。`GhosttyRuntime` がそれを `GhosttySurfaceView.receiveCommandFinished`
に転送し、`.commandFinished` イベントになったところで
`TerminalWindowController` が `TerminalAutocompleteCoordinator.handleCommandFinished`
にルーティングする。`MyTTYApp` 側で OSC ストリームをもう一度スキャンする
のではなく、アダプター境界から構造化されたアクションを受け取る形にして
あるのは、Ghostty 固有の処理を `GhosttyAdapter` の内側に留めておくためだ
(この境界がなぜ一つに絞られているかは [architecture_ja.md](architecture_ja.md)
を参照)。

## オーバーレイの位置合わせ

ゴーストテキストのラベルは、シェル自身のカーソルのちょうど右隣、最後に
入力した文字の 1 セル右に正確に重ならなければならず、フォントサイズや
ウィンドウのリサイズにも遅延なく追従する必要がある。
`GhosttySurfaceView.terminalCursorRect` はこれを libghostty から直接、
`ghostty_surface_ime_point` 経由で取得している。これは IME 変換中に候補
ウィンドウを配置するのに使っているのと同じキャレット矩形だ。テキスト
グリッドから別途カーソル位置を計算するのではなく同じ呼び出しを再利用して
いるおかげで、オートコンプリートのオーバーレイと IME 変換がカーソル位置に
ついて食い違うことは構造上起こりえない。両者は同じ座標を読んでいるからだ。

## 意図的に見送ったもの

実現可能性調査の後半の段階では、bash と fish 向けのブリッジ、tmux
サポート、Git ブランチのような動的候補、MIT ライセンスの Fig 互換補完
データベースの同梱、コマンド発見のための `--help` パースが提案されていた。
そのいずれも出荷されておらず、それは「もっと大きな実装のうち欠けている
部分」ではない。出荷された設計そのものが補完データベースを差し込む場所を
持っていない。一度に一つだけ、そのペイン自身がすでに行ったことに由来する
候補だけを出す、という作りだからだ。静的補完へ拡張するとしたら、今の
バージョンが意図的に持っていない候補パネルとキーボードナビゲーションを
新たに作ることになり、それは今日の「Local autocomplete」が指しているものとは
別の、より大きな機能になる。

コマンド履歴の学習にも意図的に狭い安全網しか張っていない。
`TerminalAutocompleteEngine` と `TerminalAutocompleteSession` はコマンド
ラインの文字列をそのまま保存するだけで、しかも終了コードが `0` のときに
限って保存する。フラグや引数に含まれる秘密情報を伏せ字にする処理は
存在しない。実際の緩和策はフィルタリングではなく範囲の狭さで、ディスクには
何も書き込まれず、リストはメモリ上で最大 200 件までに制限され、ペインを
閉じれば消える。
